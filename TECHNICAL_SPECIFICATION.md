## 1. Назначение и границы проекта

### 1.1. Что мы строим

**PHP AI Platform** — унифицированная платформа инференса AI-моделей для PHP-приложений. Это не «TensorFlow для PHP» и не библиотека обучения нейросетей. Это **мост** между зрелыми нативными движками (ONNX Runtime, llama.cpp, OpenBLAS) и удобным PHP API.

### 1.2. Ключевое позиционирование

- Единый интерфейс для инференса AI-моделей в PHP
- Мост между PHP-экосистемой и миром AI-моделей
- Удобный SDK, позволяющий PHP-разработчику использовать современные модели (LLM, embedding, vision, audio) не выходя из языка

### 1.3. Что входит в проект

| Компонент | Статус |
|---|---|
| Единый AI API для PHP | Основная задача проекта |
| ONNX Runtime инференс (CPU+GPU) | Адаптер над существующими библиотеками |
| llama.cpp инференс (CPU+GPU) | Новый биндинг через FFI |
| CPU Native инференс (RubixML/Tensor) | Адаптер над существующим расширением |
| Токенизация (BPE, WordPiece и др.) | Новый пакет |
| Embedding API | Новый пакет |
| Vector Store | Новый пакет |
| Pipeline API (Fibers-based) | Новый пакет |
| Model Hub (загрузка, кэширование, верификация) | Новый пакет |
| DataFrame API | Отложенный пакет |
| Стриминг токенов (LLM чат) | В составе llama-backend и pipeline |
| Фасадный слой для упрощённого доступа | Пакет ai |

### 1.4. Что НЕ входит в проект

- Обучение нейросетей (training)
- Autograd / автоматическое дифференцирование
- Оптимизаторы (Adam, SGD и т.д.)
- Собственные CUDA-ядра или Vulkan-шейдеры
- Компьютерное зрение как отдельный пакет (только через ONNX-модели)
- Обработка аудио как отдельный пакет (только через ONNX-модели)
- FANN (отвергнут полностью)
- Написание собственного ONNX Runtime

## 2. Архитектурные принципы

### 2.1. Фундаментальные принципы

1. **Мульти-бэкенд архитектура.** Единый PHP API над разными нативными движками. Пользователь не знает, какой бэкенд под капотом.

2. **Inference-only.** Платформа загружает готовые модели и выполняет прямой проход (forward pass). Обучение — не наша ниша.

3. **FFI-first, затем нативное расширение.** На первом этапе всё взаимодействие с нативным кодом — через PHP FFI. Переход на нативные расширения — только если FFI станет узким горлом.

4. **Максимальное переиспользование существующего.** Не писать свои матричные библиотеки, свои токенизаторы, свои векторные хранилища, если уже есть зрелые решения с C API, доступные через FFI.

5. **Пакетная архитектура со staged delivery.** 8–10 пакетов для MVP с возможностью расширения до 20+. Каждый пакет независим и имеет чёткую зону ответственности.

6. **Zero-copy там, где возможно.** Минимизация копирования данных между PHP и нативным кодом. Использование FFI-буферов напрямую.

7. **Минимальный порог входа.** Фасадный слой позволяет начать работу в одну строку. Глубокая настройка доступна через явное конфигурирование бэкендов.

### 2.2. Обязательные технологические требования

- Минимальная версия PHP: **8.5** (Pipe Operator, Clone With, NoDiscard, Closures в Constant Expressions)
- Совместимость с PHP 8.6+ (forward compatibility)
- Отсутствие жёсткой привязки к конкретному фреймворку
- Работа в окружениях: CLI, PHP-FPM, Laravel Octane, RoadRunner, FrankenPHP, Swoole
- Поддержка ОС: Linux (x86_64, aarch64), macOS (x86_64, arm64), Windows (x86_64) — через прекомпилированные бинарники

## 4. Многослойная архитектура

### 4.1. Общая схема слоёв (сверху вниз)

```
СЛОЙ 1 — USER LAND CODE
  Код конечного пользователя. Односложные вызовы.
  Не содержит знаний о бэкендах, устройствах, тензорах.

СЛОЙ 2 — FACADE / BUILDER LAYER
  Унифицированная точка входа. Роутинг по типам задач.
  Управление глобальной конфигурацией.
  Предоставляет: AI::chat(), AI::embed(), AI::classify(), AI::stream()

СЛОЙ 3 — BACKEND DRIVER LAYER
  Абстрактные интерфейсы Backend, Model, Tensor.
  Конкретные драйверы: OnnxBackend, LlamaBackend, CpuNativeBackend.
  Каждый драйвер реализует единый контракт.

СЛОЙ 4 — NATIVE RUNTIMES
  Нативные библиотеки, вызываемые через FFI.
  ONNX Runtime, llama.cpp, RubixML/Tensor (OpenBLAS).
  Не содержат PHP-кода (кроме FFI-обвязки).
```

### 4.2. Поток данных (сверху вниз)

1. Пользователь вызывает метод фасада (AI::embed, AI::chat и т.д.)
2. Фасад определяет тип задачи и выбирает подходящий бэкенд (по конфигурации или автоматически)
3. Фасад делегирует бэкенду подготовку входных данных (токенизация, препроцессинг)
4. Бэкенд загружает модель (из кэша или файловой системы)
5. Бэкенд передаёт тензоры в нативный движок через FFI
6. Нативный движок выполняет инференс (на CPU или GPU)
7. Результат возвращается обратно по цепочке: движок → бэкенд → фасад → пользователь

### 4.3. Смена бэкенда «на лету»

Глобальная конфигурация фасада позволяет сменить бэкенд в любой момент. Все последующие вызовы пойдут через новый бэкенд. Модели, загруженные в предыдущем бэкенде, остаются валидными — бэкенды изолированы друг от друга.

## 5. Слой 1: Пользовательский API (User Land)

### 5.1. Концепция

Самый верхний слой. Пользователь взаимодействует исключительно с фасадом `AI`. Никаких прямых обращений к бэкендам, тензорам, устройствам.

### 5.2. Точки входа

| Метод | Назначение | Вход | Выход |
|---|---|---|---|
| `AI::chat(array $messages): string` | Чат с LLM | Сообщения в формате ChatML | Строка ответа |
| `AI::stream(array $messages): Generator` | Стриминг токенов чата | Сообщения в формате ChatML | Generator токенов (строки) |
| `AI::embed(string\|array $input): array` | Эмбеддинг текста(ов) | Строка или массив строк | Вектор(ы) float32 |
| `AI::classify(string\|resource $input): array` | Классификация (текст/изображение) | Текст или путь/ресурс изображения | Метки с confidence |
| `AI::moderate(string $text): array` | Модерация контента | Текст | Категории и флаги |
| `AI::predict(array $features): array` | Предиктивная модель | Массив признаков | Предсказание |
| `AI::similarity(string $a, string $b): float` | Косинусное сходство | Две строки | float32 |

### 5.3. Формат ChatML

Все методы чата принимают и возвращают данные в формате ChatML (совместим с OpenAI API):
- `role`: `system`, `user`, `assistant`, `tool`
- `content`: строка или массив content-parts (для мультимодальных моделей)
- `name` (опционально): имя участника
- `tool_calls` / `tool_call_id`: для function calling

## 6. Слой 2: Фасад / Builder Layer

### 6.1. Класс `AI`

Единая точка входа для пользователя. Реализует:

- **Глобальную конфигурацию** через статический метод `AI::config(array $config)`
- **Выбор бэкенда** через `AI::backend(string $name)` — переключение активного бэкенда
- **Выбор устройства** через `AI::device(string $device)` — CPU, CUDA, Auto
- **Роутинг задач** — автоматический выбор бэкенда в зависимости от типа задачи:
  - Чат / стриминг → LlamaBackend (если есть), иначе OnnxBackend
  - Эмбеддинг → OnnxBackend
  - Классификация → OnnxBackend или CpuNativeBackend (для простых моделей)
- **Ленивую инициализацию** — бэкенд загружается только при первом обращении
- **Сброс состояния** — `AI::reset()` очищает все загруженные модели и кэши

### 6.2. Конфигурационные ключи

| Ключ | Тип | По умолчанию | Описание |
|---|---|---|---|
| `backend` | string | `auto` | Бэкенд по умолчанию: `onnx`, `llama`, `cpu`, `auto` |
| `device` | string | `auto` | Устройство: `cpu`, `cuda`, `auto` |
| `model_cache` | string | `sys_get_temp_dir() . '/ferry-ai-models'` | Директория кэширования моделей |
| `max_tokens` | int | 2048 | Максимальное число токенов для чата |
| `temperature` | float | 0.7 | Температура для LLM |
| `top_p` | float | 1.0 | Top-p сэмплирование |
| `stream_timeout` | int | 30 | Таймаут в секундах для стриминга |
| `verify_signatures` | bool | true | Проверять криптографические подписи моделей |
| `log_level` | string | `warning` | Уровень логирования |

### 6.3. Механизм Auto-выбора бэкенда

При `backend: auto` фасад проверяет доступность бэкендов в порядке приоритета:
1. Проверка наличия llama.cpp shared library → LlamaBackend
2. Проверка наличия ONNX Runtime shared library → OnnxBackend
3. Fallback → CpuNativeBackend (через RubixML/Tensor)

Для каждой конкретной задачи приоритет может переопределяться:
- Задача «чат» при наличии llama.cpp → LlamaBackend (даже если глобально выбран onnx)
- Задача «эмбеддинг» → OnnxBackend (llama.cpp неэффективен для эмбеддингов)
- Задача «классификация табличных данных» → CpuNativeBackend (меньше оверхеда)

## 7. Слой 3: Backend Driver Layer

### 7.1. Контракт `Backend`

Каждый бэкенд обязан реализовать единый интерфейс. Контракт определяет:

- **Доступные устройства** — возвращает список Device, доступных данному бэкенду (зависит от скомпилированных опций нативного движка)
- **Загрузка модели** — из файла (путь), из потока (resource), из имени на HuggingFace Hub (строка с префиксом `hf://`)
- **Выполнение инференса** — принимает массив входных данных, возвращает массив выходных
- **Интроспекция модели** — метаданные: входные/выходные имена, shapes, dtypes
- **Выгрузка модели** — освобождение нативной памяти
- **Версия бэкенда** — строковый идентификатор версии нативного движка

### 7.2. Контракт `Model`

Абстракция загруженной модели. Контракт определяет:

- **Выполнение (run)** — принимает ассоциативный массив input_name → данные, возвращает output_name → данные
- **Интроспекция входов** — имена, shapes (с динамическими осями), dtypes
- **Интроспекция выходов** — имена, shapes, dtypes
- **Метаданные** — произвольный массив: автор, лицензия, версия, описание, теги, размер
- **Подпись** — криптографическая подпись файла модели (если есть)

### 7.3. Контракт `Tensor`

Абстракция многомерного тензора. Контракт определяет:

- **Форма** — массив целых чисел (размерности)
- **Тип данных** — значение enum DType
- **Перенос между устройствами** — метод to(Device), возвращает новый тензор на целевом устройстве (без мутации исходного)
- **Экспорт в массив PHP** — для маленьких тензоров (отладка, препроцессинг)
- **Доступ к сырым данным** — FFI-буфер (для zero-copy передачи между бэкендами)
- **Арифметические операции** — сложение, умножение, транспонирование (делегируются бэкенду)
- **Индексация через ArrayAccess** — доступ к элементам как в массиве
- **Countable** — общее число элементов
- **JsonSerializable** — сериализация для REST API
- **__serialize / __unserialize** — эффективное кэширование

### 7.4. Контракт `Tokenizer`

Абстракция токенизатора. Контракт определяет:

- **Кодирование** — текст → массив token IDs
- **Декодирование** — массив token IDs → текст
- **Размер словаря** — общее число токенов
- **Специальные токены** — BOS, EOS, PAD, UNK, SEP, CLS, MASK
- **Загрузка из tokenizer.json** — стандартный формат HuggingFace
- **Пакетное кодирование** — массив текстов → массив последовательностей token IDs + attention masks
- **Чанкинг** — разбиение длинных текстов с перекрытием (для RAG)

### 7.5. Реестр бэкендов

Центральный реестр (Registry pattern), в котором регистрируются все доступные бэкенды. Фасад обращается к реестру для получения экземпляра бэкенда по имени.

Реестр поддерживает:
- Регистрацию новых бэкендов (сторонние разработчики могут добавить свой)
- Приоритеты бэкендов для разных типов задач
- Проверку доступности (наличие shared library)
- Ленивую инициализацию

## 8. Слой 4: Нативные движки (Native Runtimes)

### 8.1. Общие принципы взаимодействия через FFI

Все нативные движки загружаются через PHP FFI. Процесс:

1. PHP-пакет содержит FFI-определения структур и функций C API
2. При первом обращении FFI загружает shared library (.so/.dylib/.dll)
3. Вызовы C-функций происходят напрямую из PHP с минимальным оверхедом
4. Пакетные операции амортизируют оверхед FFI
5. Буферы данных передаются через FFI-указатели (zero-copy там, где возможно)

### 8.2. ONNX Runtime

**Нативный движок:** Microsoft ONNX Runtime  
**Тип:** универсальный движок инференса  
**Поддерживаемые GPU-бэкенды:** CUDA, TensorRT, CoreML, DirectML, OpenVINO, ROCm  
**Роль в проекте:** Основной (primary) бэкенд для всех задач, кроме LLM-чата  

**Ответственность:**
- Загрузка .onnx моделей (из PyTorch, TF, JAX, sklearn и др.)
- Выполнение инференса на CPU или GPU
- Поддержка всех стандартных ONNX-операторов
- Управление памятью устройства
- Поддержка ONNX-типов: sequences, maps, optionals

**Используемая PHP-библиотека:** `phpmlkit/onnxruntime`  
**Причина выбора:** zero-copy буферы, NDArray-интероп, прекомпилированные бинарники под платформы, полная поддержка ONNX-типов

### 8.3. llama.cpp

**Нативный движок:** llama.cpp (ggerganov)  
**Тип:** движок для Large Language Models  
**Поддерживаемые GPU-бэкенды:** CUDA, Metal, Vulkan, OpenCL  
**Роль в проекте:** Специализированный бэкенд для LLM-чата и стриминга  

**Ответственность:**
- Загрузка GGUF-моделей
- Чат-инференс (префиксное заполнение + генерация токенов)
- Стриминг токенов (генерация по одному токену)
- Управление контекстом (KV-кэш)
- Сэмплирование токенов (temperature, top-p, top-k, repetition penalty)
- Грамматики и structured output (JSON schema, GBNF)
- Function calling (через грамматики)

**PHP-интерфейс:** Новый пакет `llama-backend` через FFI. C API llama.cpp стабилен и хорошо документирован.

### 8.4. CPU Native (RubixML/Tensor)

**Нативный движок:** RubixML/Tensor (C-расширение на Zephir, поверх OpenBLAS + LAPACKE)  
**Тип:** CPU-движок линейной алгебры и классического ML  
**Роль в проекте:** Fallback-бэкенд для простых моделей и табличных данных  

**Ответственность:**
- Матричные и векторные операции на CPU (OpenBLAS)
- Классический ML: классификация, регрессия, кластеризация
- Предобработка данных (normalization, imputation, encoding)
- Fallback, когда ONNX Runtime и llama.cpp недоступны

**Примечание:** Несмотря на название «Pure PHP» в ранних обсуждениях, этот бэкенд фактически использует C-расширение (OpenBLAS + LAPACKE) и НЕ является чисто-PHP решением.

### 8.5. Матрица поддержки GPU по бэкендам

|---|---|---|---|

## 11. Пакет `tensor` — тензорная абстракция

### 11.1. Назначение

Предоставляет унифицированную работу с тензорами поверх любого бэкенда. Не реализует математику самостоятельно — делегирует бэкенду или нативному движку.

### 11.2. Основные возможности

- Создание тензора из PHP-массива (с автоматическим определением формы)
- Создание тензора заданной формы (заполнение нулями, единицами, случайными)
- Арифметические операции: сложение, вычитание, умножение (поэлементное и матричное)
- Транспонирование, изменение формы (reshape), расширение размерности
- Срезы (slicing) по осям
- Перенос между устройствами (CPU ↔ CUDA)
- Итерация по батчам (для обучения не используется, но полезно для пакетного инференса)
- Сериализация / десериализация (для кэширования препроцессированных тензоров)
- Индексация через ArrayAccess — доступ к элементам по координатам
- Countable — общее количество элементов
- Экспорт в JSON (через JsonSerializable)
- Интеграция с clone with (PHP 8.5) — создание копии на другом устройстве без мутации исходного тензора

### 11.3. Стратегия zero-copy

Тензор хранит FFI-буфер (указатель на нативную память). При передаче между бэкендами одного типа (например, два последовательных вызова ONNX Runtime) тензор не копируется — передаётся указатель. Копирование происходит только при:
- Переносе между устройствами (CPU → GPU, GPU → CPU)
- Экспорте в PHP-массив
- Передаче между разными нативными движками

### 11.4. Интеграция с бэкендами

Каждый бэкенд предоставляет свою реализацию интерфейса Tensor. Пакет `tensor` не знает о конкретных бэкендах — он работает с интерфейсом. Пользователь может создать тензор через фабрику, указав желаемый бэкенд (или auto).

## 12. Пакет `onnx-backend` — ONNX Runtime Driver

### 12.1. Назначение

Реализация интерфейса Backend через ONNX Runtime. Основной бэкенд платформы.

### 12.2. Ключевые возможности

- **Загрузка модели** из .onnx файла (локально или через поток)
- **Интроспекция** — получение списка входов/выходов, их имён, форм и типов
- **Выполнение инференса** — передача входных тензоров, получение выходных
- **Выбор провайдера исполнения** — CPU (default), CUDA, TensorRT, CoreML, DirectML, OpenVINO, ROCm
- **Настройка провайдера** — device_id, memory_limit, оптимизации графа
- **Управление памятью** — аллокация/освобождение OrtValue на конкретном устройстве
- **Поддержка ONNX-типов** — sequences (списки тензоров), maps (словари), optionals
- **Zero-copy буферы** — OrtValue как обёртка над нативной памятью (через phpmlkit/onnxruntime)
- **NDArray-интероп** — конвертация между PHP-массивами и OrtValue с минимумом копирований
- **Пакетный инференс** — несколько входов за один вызов

### 12.3. Поддерживаемые провайдеры исполнения

| Провайдер | Доступность | Приоритет |
|---|---|---|
| CPU | Всегда | Fallback |
| CUDA | Если есть NVIDIA GPU + CUDA Toolkit | Высокий |
| TensorRT | Если скомпилирован с TensorRT | Средний |
| CoreML | Только macOS | Средний (Apple Silicon) |
| DirectML | Только Windows | Низкий |
| OpenVINO | Только Intel | Низкий |
| ROCm | Только AMD | Низкий |

Приоритет auto-выбора:
1. CUDA (если доступен NVIDIA GPU)
2. CoreML (если macOS + Apple Silicon)
3. CPU

### 12.4. Оптимизации графа

ONNX Runtime поддерживает уровни оптимизации графа:
- `DISABLE_ALL` — без оптимизаций (отладка)
- `BASIC` — базовые (свёртка констант, удаление мёртвых узлов)
- `EXTENDED` — расширенные (слияние операций)
- `ALL` — все оптимизации (по умолчанию)

## 13. Пакет `llama-backend` — LLM Driver

### 13.1. Назначение

Реализация интерфейса Backend через llama.cpp. Специализированный бэкенд для больших языковых моделей (LLM).

### 13.2. Ключевые возможности

- **Загрузка GGUF-моделей** — стандартный формат llama.cpp
- **Чат-инференс** — приём сообщений в формате ChatML, генерация ответа
- **Стриминг токенов** — генерация по одному токену с возвратом через Generator (Fibers)
- **Управление параметрами генерации:**
  - temperature — температура сэмплирования
  - top_p — nucleus sampling
  - top_k — top-k sampling
  - repetition_penalty — штраф за повторения
  - frequency_penalty — штраф за частоту
  - presence_penalty — штраф за присутствие
  - max_tokens — максимальное число генерируемых токенов
  - stop — стоп-слова / стоп-токены
- **Управление контекстом:**
  - Размер контекста (n_ctx) — максимальная длина последовательности
  - Размер батча (n_batch) — для параллельной обработки промпта
  - Кэширование промпта — повторное использование KV-кэша для одинаковых префиксов (полезно для system prompt)
- **Грамматики и Structured Output:**
  - GBNF-грамматики для принудительного вывода в заданном формате
  - JSON Schema → GBNF конвертация
  - Function calling через constrained generation
- **Мультимодальность:**
  - LLaVA-подобные модели (изображение + текст)
  - Загрузка изображений, препроцессинг, передача в модель
- **Выбор GPU-бэкенда llama.cpp:**
  - CUDA (NVIDIA)
  - Metal (Apple Silicon)
  - Vulkan (кроссплатформенно)
  - OpenCL (кроссплатформенно)
- **Управление слоями на GPU:**
  - `n_gpu_layers` — сколько слоёв модели разместить на GPU (остальные на CPU)
  - Позволяет запускать большие модели, не влезающие в VRAM целиком
- **Квантизация:**
  - Поддержка всех форматов квантования llama.cpp (Q4_0, Q4_K_M, Q5_K_M, Q8_0 и др.)
  - Автоопределение типа квантизации из метаданных GGUF

### 13.3. Архитектура биндинга

Новый пакет, не основанный на существующих (все существующие PHP-биндинги к llama.cpp заброшены). Использует C API llama.cpp через FFI.

Структура пакета:
```
llama-backend/
├── src/
│   ├── LlamaBackend.php        # Реализация интерфейса Backend
│   ├── LlamaModel.php          # Реализация интерфейса Model
│   ├── FFI/
│   │   ├── LlamaCpp.php        # FFI-определения C API llama.cpp
│   │   ├── LlamaContext.php    # Обёртка над llama_context
│   │   └── LlamaBatch.php      # Обёртка над llama_batch
│   ├── Sampling/
│   │   ├── Sampler.php         # Интерфейс сэмплера
│   │   ├── GreedySampler.php   # Жадное сэмплирование
│   │   ├── TopPSampler.php     # Nucleus sampling
│   │   └── GrammarSampler.php  # Constrained generation
│   └── Grammar/
│       ├── GbnfGrammar.php     # GBNF-грамматики
│       └── JsonSchemaConverter.php  # JSON Schema → GBNF
```

### 13.4. Процесс стриминга токенов

1. Пользователь вызывает `AI::stream($messages)`
2. Фасад делегирует LlamaBackend
3. LlamaBackend токенизирует промпт
4. llama.cpp выполняет префиксное заполнение (prompt ingestion) — один проход
5. Включается цикл генерации: один вызов llama_decode() → один токен
6. Каждый токен декодируется в строку и возвращается через Generator (yield)
7. Пользователь получает токены через foreach
8. Генератор приостанавливается (Fiber suspend) между токенами — HTTP-ответ может отдавать токены через Server-Sent Events или chunked transfer

## 14. Пакет `cpu-backend` — CPU Native Driver

## 15. Пакет `tokenizer` — токенизация

### 15.1. Назначение

Унифицированный токенизатор для текстовых моделей. Работает независимо от бэкенда.

### 15.2. Ключевые возможности

- **Загрузка из tokenizer.json** — стандартный формат HuggingFace Tokenizers
- **Поддержка алгоритмов:**
  - BPE (Byte-Pair Encoding) — GPT, RoBERTa
  - WordPiece — BERT, DistilBERT
  - SentencePiece / Unigram — T5, LLaMA
- **Кодирование** — текст → массив token IDs
- **Декодирование** — массив token IDs → текст
- **Пакетное кодирование** — массив текстов → padding + attention masks
- **Специальные токены** — BOS, EOS, PAD, UNK, SEP, CLS, MASK
- **Чанкинг с перекрытием** — разбиение длинных текстов на чанки фиксированной длины с заданным overlap (для RAG-систем)
- **Подсчёт токенов** — длина текста в токенах (без полного кодирования)
- **Кэширование** — загруженный токенизатор кэшируется в OPcache shared memory между запросами

### 15.3. Стратегия реализации

Приоритет — использовать готовое решение через FFI:

1. **Приоритетный путь:** биндинг к `tokenizers-cpp` (Rust-библиотека HuggingFace Tokenizers, скомпилированная в shared library с C API). Через FFI. Даёт полную совместимость с HuggingFace tokenizer.json, высокую скорость, поддержку всех алгоритмов.

2. **Резервный путь:** Pure PHP реализация BPE. Медленнее, но не требует внешних зависимостей. Пригодна для MVP.

### 15.4. Интеграция с моделью

Токенизатор и модель связаны, но не зависят друг от друга:
- Один и тот же токенизатор (например, BERT-base) может использоваться с разными бэкендами
- Модель хранит имя токенизатора в метаданных
- Model Hub автоматически подгружает tokenizer.json вместе с моделью

## 16. Пакет `embedding` — эмбеддинги

### 16.1. Назначение

Унифицированный API для получения эмбеддингов (векторных представлений) текста, изображений и других модальностей.

### 16.2. Ключевые возможности

- **Текстовые эмбеддинги** — текст → вектор float32 (384, 768, 1024, 1536 или другая размерность)
- **Пакетные эмбеддинги** — массив текстов → массив векторов за один вызов
- **Мультиязычность** — автоматический выбор мультиязычной модели
- **Нормализация** — L2-нормализация векторов для косинусного сходства
- **Pooling strategies:**
  - CLS token (BERT-style)
  - Mean pooling (Sentence-BERT)

- **Квантованные эмбеддинги** — binary / int8 эмбеддинги для экономии места в Vector Store
- **Косинусное сходство** — сравнение двух эмбеддингов напрямую
- **Выбор модели** — встроенные модели (all-MiniLM-L6-v2 и др.) или пользовательские ONNX

### 16.3. Встроенные модели

Платформа предоставляет несколько предварительно сконвертированных ONNX-моделей для эмбеддингов:

| all-mpnet-base-v2 | 768 | ~420 MB | EN |
| multilingual-e5-small | 384 | ~470 MB | 100+ |
| bge-small-en | 384 | ~130 MB | EN |

Модели скачиваются автоматически при первом использовании через Model Hub.

### 16.4. Интеграция

Пакет `embedding` зависит от:
- `core` — интерфейсы
- `onnx-backend` — выполнение инференса (эмбеддинг-модели — это ONNX-модели)
- `tokenizer` — токенизация текста перед эмбеддингом

## 17. Пакет `pipeline` — конвейеры обработки

### 17.1. Назначение

Создание композируемых конвейеров обработки данных с использованием Fibers для неблокирующего выполнения и Pipe Operator (PHP 8.5) для декларативного синтаксиса.

### 17.2. Ключевые возможности

- **Стадии (Stages)** — каждая стадия принимает данные от предыдущей и передаёт следующей
- **Fibers-based выполнение** — каждая стадия в отдельном Fiber, приостановка и возобновление
- **Pipe Operator интеграция** — декларативная композиция стадий через `|>`
- **Генераторная модель** — пайплайн возвращает Generator, каждый элемент обрабатывается лениво
- **Параллельное выполнение** — несколько пайплайнов могут работать одновременно (через Fibers)
- **Батчевая обработка** — группировка элементов в батчи для эффективного инференса
- **Встроенные стадии:**
  - TokenizeStage — токенизация текста
  - EmbedStage — получение эмбеддингов
  - NormalizeStage — L2-нормализация
  - StoreStage — сохранение в Vector Store
  - ClassifyStage — классификация
  - FilterStage — фильтрация результатов
  - TransformStage — пользовательская трансформация (callable)
- **Обработка ошибок** — каждая стадия может выбросить исключение, пайплайн останавливается
- **Логирование** — каждая стадия может логировать вход/выход/время выполнения
- **Профилирование** — замер времени каждой стадии

### 17.3. Пример композиции (концептуально, без кода)

Пайплайн для индексации документа в RAG-системе:
1. Разбиение на чанки (ChunkStage)
2. Токенизация каждого чанка (TokenizeStage)
3. Эмбеддинг каждого чанка (EmbedStage)
4. Нормализация (NormalizeStage)
5. Сохранение в Vector Store (StoreStage)

Все стадии соединены через Pipe Operator, выполняются лениво через Fibers.

## 18. Пакет `model-hub` — управление моделями

### 18.1. Назначение

Централизованное управление жизненным циклом моделей: загрузка, кэширование, верификация, интроспекция, удаление.

### 18.2. Ключевые возможности

- **Загрузка из HuggingFace Hub** — по имени модели (например, `hf://sentence-transformers/all-MiniLM-L6-v2`)
- **Загрузка из локального файла** — путь к .onnx, .gguf, .safetensors и др.
- **Загрузка из URL** — прямая ссылка на файл модели
- **Загрузка из потока** — resource (например, из HTTP-запроса или S3)
- **Кэширование:**
  - Локальный кэш в файловой системе (настраиваемая директория)
  - Проверка целостности по SHA-256
  - LRU-вытеснение (настраиваемый лимит размера кэша)
  - Инвалидация по версии
- **Верификация:**
  - Проверка SHA-256 хэша
  - Проверка криптографической подписи (Sodium, Ed25519) — для моделей из недоверенных источников
  - Проверка формата файла (magic bytes)
- **Интроспекция модели:**
  - Без полной загрузки — чтение метаданных из заголовка
  - Формат, размер, количество параметров, архитектура
  - Требуемые входы/выходы
- **Форматы моделей:**
  - .onnx — ONNX Runtime
  - .gguf — llama.cpp
  - .safetensors — безопасный формат весов
  - .ai — проприетарный архивный формат (zip с model + tokenizer + config)
  - .rbm — RubixML model format
- **Управление версиями:**
  - Загрузка конкретной версии модели
  - Обновление модели
  - Откат к предыдущей версии
- **Очистка кэша** — удаление старых версий, неиспользуемых моделей
- **Прогрев кэша** — предзагрузка моделей при деплое

### 18.3. Интеграция с HuggingFace Hub

Использует пакет `codewithkyrian/huggingface-php` как базовый HTTP-клиент для HF API. Добавляет:
- Кэширование на уровне файловой системы
- Верификацию целостности
- Поддержку HF API token для приватных моделей
- Ретраи при сетевых ошибках
- Прогресс-бар загрузки (через Generator)
- Возобновление прерванных загрузок (Range-запросы)

## 19. Пакет `vector` — векторное хранилище

### 19.1. Назначение

Локальное векторное хранилище для семантического поиска. Основа для RAG-систем.

### 19.2. Ключевые возможности

- **Добавление векторов** — вектор + метаданные (JSON) + ID
- **Поиск похожих** — query-вектор + top-k + фильтр по метаданным
- **Удаление** — по ID или по фильтру
- **Обновление** — замена вектора и метаданных по ID
- **Фильтрация по метаданным** — WHERE-подобный синтаксис для JSON-полей
- **Метрики сходства:**
  - Косинусное сходство (cosine)
  - L2-расстояние (euclidean)
  - Скалярное произведение (dot product)
- **Индексация:**
  - HNSW (Hierarchical Navigable Small World) — через sqlite-vec
  - IVF (Inverted File) — через sqlite-vec
  - Flat (brute force) — для маленьких коллекций
- **Квантование векторов:**
  - Float32 (полная точность)
  - Float16 (половинная точность)
  - Int8 (скалярное квантование)
  - Binary (бинарные эмбеддинги)
- **Персистентность:**
  - SQLite-файл — портативный, не требует сервера
  - Возможность копирования файла между серверами
  - WAL-режим для конкурентного доступа
- **Батчевые операции** — массовое добавление/удаление векторов
- **Экспорт/Импорт** — выгрузка всей коллекции в JSON/Parquet
- **Размерность векторов** — любая (64–4096 типично)

### 19.3. Техническая реализация

Два слоя:

1. **Слой доступа (PHP):** PDO-обёртка над SQLite. Чистый PHP, без внешних зависимостей. Управление коллекциями, метаданными, базовые операции.

2. **Слой индексации (расширение SQLite):** `sqlite-vec` extension. Загружается через FFI или как SQLite extension. Обеспечивает ANN-поиск (приближённый поиск ближайших соседей).

При отсутствии sqlite-vec — fallback на brute force поиск средствами PHP (для коллекций до ~10,000 векторов).

### 19.4. Коллекции

Векторы группируются в именованные коллекции (аналогично таблицам в БД). Каждая коллекция:
- Имеет фиксированную размерность векторов
- Имеет схему метаданных (JSON Schema, опционально)
- Может иметь несколько индексов с разными метриками
- Поддерживает тэги и описание

## 22. Система типов и перечислений

### 22.1. Device (устройство вычислений)

String-backed enum. Значения:

- `CPU` — центральный процессор. Доступен всегда.
- `CUDA` — NVIDIA GPU (через CUDA). Требует CUDA Toolkit.
- `METAL` — Apple GPU (через Metal Performance Shaders). Только macOS.
- `AUTO` — автоматический выбор лучшего доступного устройства.

Правила auto-выбора:
1. Если есть NVIDIA GPU + CUDA → CUDA
2. Если macOS + Apple Silicon → METAL
3. Иначе → CPU

### 22.2. DType (тип данных тензора)

String-backed enum. Значения:

- `Float32` — 32-битное число с плавающей точкой (4 байта). Основной тип.
- `Float16` — 16-битное число с плавающей точкой (2 байта). Для GPU-инференса.
- `Int32` — 32-битное целое (4 байта). Индексы, token IDs.
- `Int64` — 64-битное целое (8 байт). Большие индексы.
- `String` — строковой тип. Для NLP-моделей с текстовыми входами.

Каждому значению соответствует метод `sizeInBytes()`.

### 22.3. BackendType (тип бэкенда)

String-backed enum. Значения:

- `Onnx` — ONNX Runtime
- `Llama` — llama.cpp
- `CpuNative` — RubixML/Tensor (CPU)

### 22.4. TokenizerType (тип токенизатора)

String-backed enum. Значения:

- `BPE` — Byte-Pair Encoding (GPT, RoBERTa)
- `WordPiece` — WordPiece (BERT)
- `SentencePiece` — SentencePiece / Unigram (T5, LLaMA)
- `Unigram` — Unigram LM (альтернативный SentencePiece)

### 22.5. Shape (форма тензора)

Readonly value object. Хранит массив целых чисел (размерности). Инвариант: все значения ≥ 0.

Методы:
- `rank()` — количество осей (длина массива)
- `size()` — общее количество элементов (произведение размерностей)
- `dimension(int $axis)` — размер по оси
- Валидация совместимости (broadcasting rules)

## 23. Управление устройствами и памятью

### 23.1. Обнаружение устройств

При инициализации бэкенда происходит обнаружение доступных устройств:

**ONNX Runtime:**
- Получение списка доступных Execution Providers
- Проверка наличия GPU через CUDA API (cudaGetDeviceCount)
- Проверка наличия CoreML (только macOS)
- Проверка доступности DirectML (только Windows)

**llama.cpp:**
- Проверка наличия CUDA-сборки llama.cpp
- Проверка наличия Metal-сборки (macOS)
- Проверка наличия Vulkan-сборки
- Проверка количества GPU-слоёв, доступных для оффлоада

**CPU Native:**
- Только CPU. Проверка наличия OpenBLAS и количества потоков.

### 23.2. Перенос тензоров между устройствами

Операция `tensor.to(Device::CUDA)` создаёт новый тензор на целевом устройстве. Исходный тензор не мутируется (иммутабельность).

Этапы:
1. Проверка, доступно ли целевое устройство
2. Если тензор уже на целевом устройстве — возврат this (без копирования)
3. Аллокация памяти на целевом устройстве
4. Копирование данных
5. Освобождение памяти исходного тензора (если он больше не нужен — через WeakMap / destructor)

### 23.3. Управление памятью

- Каждый бэкенд управляет своей памятью (CPU или GPU)
- PHP garbage collector освобождает PHP-объекты
- Деструкторы PHP-объектов освобождают нативную память через FFI-вызовы (cudaFree, OrtReleaseValue и т.д.)
- WeakMap используется для отслеживания градиентов/графов без создания циклических ссылок

### 23.4. Лимиты памяти

- Настраиваемый лимит VRAM для GPU-инференса (чтобы не забить всю память GPU)
- Настраиваемый лимит RAM для CPU-инференса
- При превышении лимита — исключение InferenceException
- Для llama.cpp: возможность разместить часть слоёв на CPU, часть на GPU (n_gpu_layers)

## 24. Стриминг и асинхронность

### 24.1. Стриминг токенов (LLM чат)

Сценарий: пользователь отправляет сообщение в чат, ответ генерируется токен за токеном и отображается в реальном времени.

Реализация:
1. Метод `AI::stream($messages)` возвращает Generator
2. Generator внутри использует Fiber
3. Fiber вызывает llama_decode() в цикле для каждого токена
4. Между токенами Fiber приостанавливается
5. HTTP-контроллер (Laravel/Symfony) получает токены через foreach
6. Контроллер отправляет каждый токен клиенту через:
   - Server-Sent Events (text/event-stream)
   - Chunked Transfer Encoding
   - WebSocket (через RoadRunner/Swoole/FrankenPHP)

### 24.2. Pipeline через Fibers

Каждая стадия пайплайна выполняется в отдельном Fiber. Это позволяет:
- Приостанавливать выполнение пайплайна между элементами
- Параллельное выполнение нескольких пайплайнов
- Неблокирующую обработку (пока одна стадия ждёт FFI-вызов, другие могут работать)

### 24.3. Таймауты

- Настраиваемый таймаут для стриминга (по умолчанию 30 секунд)
- При превышении таймаута — Fiber прерывается, выбрасывается исключение
- Частичный результат (уже сгенерированные токены) доступен в исключении

### 24.4. Отмена стриминга

Пользователь может прервать стриминг:
- Клиент закрывает соединение
- PHP определяет `connection_aborted()`
- Fiber завершается, нативная память освобождается

## 25. Безопасность

### 25.1. Верификация моделей

Каждая загружаемая модель проходит верификацию:

1. **SHA-256 хэш** — проверка целостности файла. Хэш хранится в метаданных модели или в отдельном .sha256 файле.

2. **Криптографическая подпись (Ed25519)** — для моделей из недоверенных источников. Подпись проверяется через Sodium extension. Формат: `weights.bin + signature` — подпись хранится отдельно или в метаданных.

3. **Magic bytes** — проверка, что файл действительно является моделью заявленного формата (первые байты .onnx, .gguf, .safetensors и т.д.).

4. **Опциональность** — верификация может быть отключена через конфигурацию (не рекомендуется для production).

### 25.2. Безопасная загрузка из недоверенных источников

- Все загрузки через HTTPS
- Проверка SSL-сертификатов
- Ограничение размера загружаемого файла (настраиваемый лимит)
- Проверка MIME-типа (fileinfo)
- Загрузка во временную директорию, затем атомарное перемещение в кэш
- Очистка временных файлов при ошибке

### 25.3. Изоляция моделей

- Каждая модель загружается в свою область памяти
- Модели не имеют доступа к файловой системе (кроме своего файла)
- Модели не имеют сетевого доступа
- Нативные вызовы через FFI ограничены API движка (нет произвольного кода)

### 25.4. FFI безопасность

- FFI-функции загружаются только из доверенных shared libraries
- Пути к библиотекам проверяются на предмет traversal attacks
- `ffi.enable=preload` — библиотеки загружаются только при старте PHP (нельзя загрузить произвольную .so во время выполнения)

### 25.5. Конфиденциальность данных

- Пользовательские данные (тексты, изображения) не логируются по умолчанию
- Данные передаются в нативный движок в оперативной памяти, не записываются на диск
- Для отладки можно включить логирование (с предупреждением в документации)

## 26. Кэширование и производительность

### 26.1. Уровни кэширования

| Уровень | Что кэшируется | Где | Время жизни |
|---|---|---|---|
| L1: OPcache shared memory | Токенизаторы, словари, конфиги моделей | RAM | До перезапуска PHP |
| L2: Файловый кэш | Загруженные модели (.onnx, .gguf) | Диск | LRU, настраиваемый лимит |
| L3: Препроцессированные тензоры | Результаты токенизации, нормализации | Диск (сериализация) | По ключу кэша (sha256 входа) |
| L4: Shared Memory (shmop) | Веса моделей, разделяемые между процессами | RAM (System V) | До перезапуска ОС |

### 26.2. Стратегия кэширования моделей

- При первом использовании модель скачивается из HuggingFace Hub
- Сохраняется в `model_cache` директорию
- При повторном использовании — загружается из кэша
- Проверка целостности по SHA-256 при каждой загрузке (опционально)
- LRU-вытеснение: при превышении лимита размера кэша удаляются наименее используемые модели
- Инвалидация: модель проверяется на наличие обновлений в HuggingFace Hub (опционально, по тегу/версии)

### 26.3. Оптимизации инференса

- **Пакетный инференс** — несколько входов за один вызов модели (утилизация GPU)
- **Пул моделей** — предзагруженные модели не выгружаются между запросами
- **Прогрев** — предзагрузка моделей при старте приложения (через `AI::warmup(['model1', 'model2'])`)
- **Graph optimization** — ONNX Runtime оптимизирует граф перед выполнением
- **FP16 инференс** — использование половинной точности на GPU (быстрее, меньше VRAM)
- **KV-кэш (llama.cpp)** — кэширование ключей и значений для ускорения последовательной генерации
- **Prompt caching (llama.cpp)** — повторное использование KV-кэша для одинаковых префиксов (system prompt)

### 26.4. Потоковая загрузка больших моделей

Модели размером 18+ GB (LLaMA 70B, Mixtral и т.д.) не загружаются в память PHP целиком. Используется:
- `fopen()` и стриминг в нативный движок чанками
- `mmap()` через FFI — отображение файла в память (поддерживается llama.cpp)
- Memory-mapped I/O — модель загружается в виртуальное адресное пространство, страницы подгружаются по требованию

Проприетарный формат для распространения моделей. ZIP-архив, содержащий:

## 28. Межпроцессное взаимодействие

### 28.1. Разделяемая память (Shared Memory)

Несколько PHP-воркеров (FPM, Octane, RoadRunner) могут разделять одни и те же веса моделей в памяти:

**Технология:** System V Shared Memory (shmop extension) для CPU, CUDA IPC для GPU.

**Сценарий:**
1. Первый воркер загружает модель в shared memory
2. Последующие воркеры подключаются к тому же сегменту shared memory
3. Инференс выполняется каждым воркером независимо (read-only веса)
4. Экономия RAM: 1 копия весов вместо N копий для N воркеров

**Применимость:**
- ONNX Runtime: не поддерживает shared memory напрямую. Требуется отдельный inference server.
- llama.cpp: поддерживает mmap. Несколько процессов могут разделять один mmap-файл.
- RubixML/Tensor: не поддерживает. Каждый процесс загружает свою копию.

### 28.2. Асинхронный инференс

В долгоживущих процессах (RoadRunner, FrankenPHP, Swoole, Octane):
- Инференс выполняется в отдельном Fiber, не блокируя основной цикл
- Несколько инференсов могут выполняться параллельно (если бэкенд поддерживает)
- Результаты возвращаются через Promise/Future (если доступно) или Generator

### 28.3. Блокировки и конкурентный доступ

- Модели read-only — не требуют блокировок
- Кэш моделей — блокировка на запись (flock) при скачивании, без блокировок на чтение
- Vector Store — SQLite WAL-режим для конкурентного доступа (много читателей, один писатель)
- Shared Memory — не требует блокировок (read-only веса)

## 29. Конфигурация и управление состоянием

### 29.1. Глобальная конфигурация

Единая точка конфигурации через `AI::config([])`. Вызывается один раз при старте приложения (в bootstrap или service provider).

Конфигурация хранится в статическом свойстве и доступна всем компонентам платформы.

### 29.2. Конфигурация бэкендов

Каждый бэкенд может иметь свою специфическую конфигурацию, передаваемую через вложенный массив:

```
AI::config([
    'backend' => 'llama',
    'backends' => [
        'llama' => [
            'model_path' => '/models/mistral-7b.Q4_K_M.gguf',
            'n_ctx' => 8192,
            'n_gpu_layers' => 33,
        ],
        'onnx' => [
            'providers' => ['CUDA', 'CPU'],
            'graph_optimization_level' => 'ALL',
        ],
    ],
]);
```

### 29.3. Состояние и его сброс

- `AI::reset()` — сбрасывает всю конфигурацию, выгружает все модели, очищает кэши
- `AI::resetBackend('llama')` — сбрасывает только конкретный бэкенд
- `AI::resetModel('mistral-7b')` — выгружает конкретную модель

### 29.4. Переменные окружения

Альтернативный способ конфигурации — через переменные окружения:

## 31. Use Cases и сценарии использования

### 31.1. RAG-система на PHP

**Сценарий:** пользователь задаёт вопрос, система ищет релевантные документы и генерирует ответ с учётом контекста.

**Задействованные пакеты:** ai, embedding, vector, pipeline, tokenizer, llama-backend

**Поток:**
1. Документы индексируются: chunk → embed → vector store (через pipeline)
2. При запросе: query → embed → search vector store → retrieve top-k chunks
3. Формируется промпт: system prompt + retrieved chunks + user query
4. LLM генерирует ответ со ссылками на источники
5. Ответ возвращается пользователю (или стримится)

### 31.2. Семантический поиск по товарам

**Сценарий:** e-commerce каталог с поиском по смыслу (не по ключевым словам).

**Задействованные пакеты:** ai, embedding, vector

**Поток:**
1. Описания товаров → embed → vector store
2. Поисковый запрос → embed → search vector store
3. Возвращаются похожие товары с метаданными (цена, категория, наличие)

### 31.3. Модерация контента

**Сценарий:** пользовательский контент проверяется на violations (спам, оскорбления, NSFW).

**Задействованные пакеты:** ai, onnx-backend

**Поток:**
1. Текст/изображение → classify (модель модерации)
2. Возвращаются категории нарушений и confidence scores
3. При превышении порога — контент блокируется

### 31.4. Чат-бот в Laravel

**Сценарий:** типичное Laravel-приложение с чат-ботом на базе LLM.

**Задействованные пакеты:** ai, llama-backend, tokenizer

**Поток:**
1. В контроллере: `AI::stream($messages)`
2. Токены возвращаются через Generator
3. Контроллер отправляет SSE клиенту
4. На фронтенде токены отображаются в реальном времени

### 31.5. Классификация обращений в техподдержку

**Сценарий:** входящие тикеты автоматически классифицируются по категориям.

**Задействованные пакеты:** ai, onnx-backend (или cpu-backend для простых моделей)

**Поток:**
1. Текст тикета → classify
2. Возвращается категория и confidence
3. Тикет маршрутизируется в нужный отдел

## 32. Что за границами проекта

### 32.1. Категорически НЕ делаем

| Компонент | Причина |
|---|---|
| Обучение нейросетей (training) | PHP не конкурент Python. Огромный объём работы. |
| Autograd / автоматическое дифференцирование | Чистый PHP — слишком медленно. C-расширение — слишком сложно. |
| Оптимизаторы (SGD, Adam, etc.) | Нужны только для обучения. |
| Собственные CUDA-ядра | Существующие движки уже решают эту задачу. |
| FANN (Fast Artificial Neural Network) | Устарел (2012), только MLP, нет GPU, не поддерживается. |
| Собственный ONNX Runtime | Переизобретение Microsoft-продукта. |
| Собственный формат моделей | ONNX и GGUF — индустриальные стандарты. |
| Тренировочные датасеты и их загрузчики | Не наша ниша. |
| Визуализация графов моделей | Второстепенно. |
| AutoML / подбор гиперпараметров | Не наша ниша. |

### 32.2. Отложено (YAGNI для сейчас)

| Компонент | Когда может понадобиться |
|---|---|
| DataFrame API | Когда будет спрос на табличную предобработку |
| Computer Vision (отдельный пакет) | Когда появится спрос на CV-специфичные операции |
| Audio Processing (отдельный пакет) | Когда появится спрос на Audio-специфичные операции |
| Phar-упаковка моделей | Креативно, но непрактично для multi-GB моделей |
| Многопроцессное обучение (pcntl) | Выходит за рамки inference-ниши |
| GMP (Bloom Filter / BitSet) | Нишевый сценарий, не для MVP |
| DOM/XML парсинг ONNX | Перекрывается самим ONNX-бэкендом |

## 34. Реестр рисков

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| llama.cpp C API изменится | Средняя | Высокое | Использовать стабильные версии (b4000+). Фиксировать версию в зависимостях. CI-тесты на совместимость. |
| ONNX Runtime изменит C API | Низкая | Высокое | Microsoft поддерживает обратную совместимость. Версионирование через phpmlkit/onnxruntime. |
| FFI отключён на целевом хостинге | Средняя | Критичное | Документировать требования. Предлагать VPS/ dedicated. CPU-backend как fallback без FFI. |
| Производительность FFI недостаточна | Низкая | Среднее | Пакетные вычисления амортизируют оверхед. При необходимости — переход на нативное расширение. |
| RubixML/Tensor сложно скомпилировать | Высокая | Низкое | Это fallback-бэкенд. Основные бэкенды — ONNX и llama.cpp. RubixML/Tensor — опциональная зависимость. |
| Конкуренция с Python-экосистемой | Низкая | Низкое | Мы не конкурируем в обучении. Наша ниша — inference из PHP. Это другой рынок. |
| Мало звёзд у phpmlkit/onnxruntime (⭐8) | Средняя | Среднее | Резервный вариант — ankane/onnxruntime-php (⭐149). Обе библиотеки используют одно C API. Миграция между ними — замена адаптера. |
| Утечки памяти в FFI | Средняя | Высокое | Тщательное тестирование. Деструкторы PHP-объектов → освобождение нативных ресурсов. Valgrind / LeakSanitizer. |
| Большие модели не влезают в VRAM | Средняя | Среднее | FP16, квантование, offloading слоёв на CPU (llama.cpp n_gpu_layers). |
| Сложность отладки FFI-вызовов | Средняя | Среднее | Логирование, бэктрайсы в PHP 8.5 fatal errors. Отдельный debug-режим. |

## 35. Глоссарий

| Термин | Определение |
|---|---|
| **Инференс (Inference)** | Выполнение прямого прохода (forward pass) модели для получения предсказания. Без изменения весов. |
| **Бэкенд (Backend)** | Конкретная реализация интерфейса Backend, использующая определённый нативный движок (ONNX Runtime, llama.cpp, RubixML/Tensor). |
| **Драйвер (Driver)** | Синоним бэкенда в контексте паттерна Driver Pattern. |
| **Фасад (Facade)** | Класс AI, предоставляющий упрощённый статический интерфейс ко всей платформе. |
| **ONNX (Open Neural Network Exchange)** | Открытый формат для представления моделей машинного обучения. Конвертируется из PyTorch, TensorFlow, JAX и др. |
| **ONNX Runtime** | Нативный движок от Microsoft для выполнения ONNX-моделей. Поддерживает CPU, CUDA, TensorRT, CoreML, DirectML, OpenVINO, ROCm. |
| **llama.cpp** | Нативный движок для запуска больших языковых моделей (LLM). Поддерживает GGUF-формат, квантование, GPU-ускорение. |
| **GGUF** | Формат файлов моделей для llama.cpp. Содержит веса, конфигурацию и метаданные. |
| **FFI (Foreign Function Interface)** | Механизм PHP для вызова функций из нативных shared libraries (.so/.dylib/.dll) напрямую из PHP-кода. |
| **Zero-copy** | Техника передачи данных между PHP и нативным кодом без копирования. Используются FFI-указатели на shared память. |
| **Fibers** | Механизм PHP 8.1 для кооперативной многозадачности. Позволяет приостанавливать и возобновлять выполнение функции. |
| **Pipe Operator `\|>`** | Оператор PHP 8.5 для цепочечного вызова функций: `$a \|> f(...) \|> g(...)`. |
| **Clone With** | Синтаксис PHP 8.5 для создания клона объекта с изменёнными свойствами: `clone($obj, ['prop' => 'new'])`. |
| **ChatML** | Формат сообщений для чат-моделей (совместим с OpenAI API). Роли: system, user, assistant, tool. |
| **Эмбеддинг (Embedding)** | Векторное представление текста/изображения/звука. Используется для семантического поиска. |
| **Vector Store** | База данных для хранения и поиска векторов (эмбеддингов). |
| **RAG (Retrieval-Augmented Generation)** | Архитектура: поиск релевантных документов + генерация ответа LLM с учётом найденного контекста. |
| **KV-кэш (Key-Value Cache)** | Кэш ключей и значений attention-слоёв в трансформере. Ускоряет последовательную генерацию токенов. |
| **Квантизация (Quantization)** | Сжатие весов модели (например, float32 → int4) для уменьшения размера и ускорения инференса. |
| **GGUF** | Формат файла модели для llama.cpp. Содержит веса + конфигурацию. Поддерживает множество форматов квантования. |
| **HuggingFace Hub** | Центральный репозиторий AI-моделей. Источник моделей для Model Hub. |
| **tokenizer.json** | Стандартный формат конфигурации токенизатора от HuggingFace Tokenizers. |
| **GBNF (GGML BNF)** | Формат грамматик для constrained generation в llama.cpp. |
| **Structured Output** | Генерация текста, который гарантированно соответствует заданной схеме (JSON, GBNF). |
| **HNSW (Hierarchical Navigable Small World)** | Алгоритм приближённого поиска ближайших соседей (ANN). Используется в sqlite-vec. |
| **sqlite-vec** | SQLite extension для векторного поиска. |
